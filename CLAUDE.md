## Project Overview

Aldawood Court Booking is a monorepo for a football court booking system. A Python AI agent handles customer conversations, calls a TypeScript backend for business logic, and a Next.js admin portal provides operational views. The project is currently a scaffold with stubbed implementations.

## Architecture

Three services communicate in a pipeline:

1. **apps/agent** (Python, FastAPI + LangGraph) — Conversational booking agent. Classifies intent, extracts entities, and orchestrates a multi-node LangGraph state machine (`app/graph/builder.py`). Calls the backend API via `app/clients/backend_client.py` for all data mutations. Uses OpenAI for LLM calls.
2. **apps/api** (TypeScript, NestJS + Prisma + PostgreSQL) — System of record. Owns all booking logic, policy enforcement, availability checks, and pricing. Modules: auth, admins, courts, customers, bookings, pricing, availability, conversations, event-packages, policies, audit-logs, internal-agent.
3. **apps/admin-web** (TypeScript, Next.js 15, React 18) — Admin dashboard with pages for bookings, courts, conversations, packages, policies, admins, audit logs, and an agent test console.

**Key principle:** Business logic lives in the backend, the agent handles only orchestration and dialogue.

### Shared packages (npm workspaces)
- `packages/shared-types` — Shared TypeScript types
- `packages/api-contracts` — API contract definitions

## Commands

### Setup
```bash
docker compose up -d          # Start PostgreSQL (port 5432)
npm install                   # Install all Node workspace deps
pip install -r apps/agent/requirements.txt  # Python deps
cp .env.example .env          # Then fill in real values
```

### Development
```bash
npm run dev:api       # NestJS backend (watch mode, port 4000)
npm run dev:agent     # Python agent via uvicorn (reload, port 8000)
npm run dev:admin     # Next.js admin portal (port 3000)
```

### Build & Lint
```bash
npm run build:api     # nest build
npm run build:admin   # next build
npm run build         # Both
npm run lint:api      # tsc --noEmit (API)
npm run lint:admin    # tsc --noEmit (admin-web)
npm run lint          # Both
```

### Database (run from apps/api)
```bash
npm --workspace @aldawood/api run prisma:generate   # Generate Prisma client
npm --workspace @aldawood/api run prisma:migrate     # Run migrations (dev)
npm --workspace @aldawood/api run prisma:seed        # Seed data
```

## Key Files

- `apps/api/prisma/schema.prisma` — Full data model (courts, customers, bookings, events, pricing rules, conversations, policies, admin users, audit logs)
- `apps/api/src/app.module.ts` — NestJS module registry
- `apps/agent/app/graph/builder.py` — LangGraph state machine definition with node routing
- `apps/agent/app/graph/state.py` — Agent state shape
- `apps/agent/app/graph/nodes/` — Individual graph nodes (classify_intent, extract_entities, check_availability, create_booking, etc.)
- `docker-compose.yml` — PostgreSQL 16 service

## Environment Variables

Defined in `.env.example`: `DATABASE_URL`, `JWT_SECRET`, `OPENAI_API_KEY`, `API_PORT` (4000), `AGENT_PORT` (8000), `BACKEND_API_URL`, `NEXT_PUBLIC_API_BASE_URL`, `NEXT_PUBLIC_AGENT_BASE_URL`.

## Conventions

- Prisma models use `@map()` for snake_case DB column names; TypeScript uses camelCase.
- All DB primary keys are UUID v4, generated by PostgreSQL (`gen_random_uuid()`).
- All timestamps use `Timestamptz`.
- The agent graph flow: load_context → persist_user_message → classify_intent → extract_entities → validate_request → decide_next_step → (conditional routing) → persist_assistant_message → END.
- Bilingual support (Arabic/English) — models have `name`/`nameAr` field pairs, customers have `preferredLang`.
- Timezone is `Asia/Amman` by default (configurable in `BookingPolicy`).
